# 阶段 1：稳定现有服务

## 目标

先把当前企业微信到 Craft 的主链路修到可维护、可验证、可部署的状态，为后续抽象多目标适配器打基础。

## 任务清单

- [x] 修复 `/craft/save` 接口未传 `link_id`、`document_id`、`document_token` 的问题。
- [x] 统一消息拉取游标逻辑，移除对不存在的 `wecom_messages` 表的依赖。
- [x] 将回调触发拉取和后台轮询共用同一套 WeCom 消息解析函数。
- [x] 为 `unified_messages` 增加 `(source, msg_id)` 唯一约束，避免重复消息入库。
- [x] 增加投递状态记录，区分“已入库”“已投递”“投递失败”“待重试”。
- [x] 补齐企业微信回调验签和加密消息处理。
- [x] 统一默认端口、数据库路径、容器名、镜像名和日志名。
- [x] 将 Craft HTTP 调用改成非阻塞实现，避免 async 函数内部阻塞事件循环。
- [x] 清理敏感日志，避免 token、secret、私钥和完整原始消息泄露。
- [x] 为 parser、formatter、数据库去重、Craft adapter 增加单元测试。
- [x] 使用 `./docker-deploy.sh` 完成 Docker 构建、启动、健康检查和日志检查。

## 验收标准

- [x] `curl /`、`curl /openapi.json` 正常返回。
- [x] Docker 容器启动后没有新的 traceback 或错误日志。
- [x] 文本、链接、图片、文件、视频消息至少有可复现的解析和格式化测试。
- [x] 同一条消息重复处理不会重复写入存档表。
- [x] Craft 目标失败时能记录失败原因，并支持后续重试。
