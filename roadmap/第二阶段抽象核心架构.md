# 阶段 2：抽象核心架构

## 目标

把项目从“企业微信到 Craft 的单一路径”升级为“消息源到笔记目标”的通用架构，让 WeCom 和 Craft 都成为可替换模块。

## 任务清单

- [x] 定义 `SourceConnector` 接口，负责拉取消息、解析游标和下载附件。
- [x] 定义 `TargetAdapter` 接口，负责验证目标配置和投递标准化消息。
- [x] 定义统一消息模型，覆盖文本、链接、图片、视频、文件、语音和未知类型。
- [x] 定义统一附件模型，保存本地路径、文件名、MIME、大小、hash 和外部 URL。
- [x] 引入 `Router`，按发送者、群聊、消息类型、规则配置选择目标。
- [x] 引入 `Delivery` 模型，记录每条消息到每个目标的投递状态。
- [x] 将 WeCom SDK 逻辑迁移为 `sources/wecom`。
- [x] 将 Craft 写入逻辑迁移为 `targets/craft`。
- [x] 将 COS 上传逻辑改为可选的 asset storage 层。
- [x] 建立数据库 migration 机制，避免靠启动时直接执行零散 SQL。
- [x] 为核心接口提供 mock 实现，便于单元测试和示例运行。

## 验收标准

- [x] 主业务链路只依赖抽象接口，不直接依赖 Craft 服务实现。
- [x] 新增一个目标适配器不需要修改 WeCom source 代码。
- [x] 新增一个消息源不需要修改 Craft target 代码。
- [x] 路由、投递、去重、游标都有独立测试。
- [x] 旧 Craft 链路在新架构下仍能通过 Docker 验证。
